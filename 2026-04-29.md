# Architecture & Code Quality Report

### 1. Frontend Architecture Overview
Your frontend (`pr` directory) is built on a standard modern stack: **Vite + React + TypeScript + Tailwind CSS**, heavily utilizing Radix UI primitives (likely via `shadcn/ui`). 

**The HTML Entry Points:**
*   **`index.html`**: This is the primary entry point for the Vite React application (the public portfolio). Vite injects `src/main.tsx` into this file, which hydrates your React tree and renders the `dist` output for production.
*   **`video-manager.html` / `manage-videos.html`**: These are *not* part of the Vite build pipeline. They are static files served exclusively by your local Node Server (`video-manager-server.js` on port 3000) as the UI for your local CMS.

**The Build Pipeline (`vite.config.ts`)**:
You have a well-optimized configuration. You are correctly splitting your production build into manual chunks (`vendor`, `animations`, `ui`), which is excellent for caching. The frontend consumes its data statically—meaning every time a video is added, the site requires a Git push and an external CI/CD rebuild (likely via GitHub Pages/Actions).

### 2. Full-Stack Video Management Flow
Your application's data flow is highly unconventional but effective for a static site. It forms a self-modifying codebase.

Here is the life cycle when you manage videos:
1.  **The Launcher:** `manager-app/video_manager.py` acts as a desktop GUI. It verifies your local Node environment, installs packages, and spawns `node video-manager-server.js` as a background process.
2.  **The Local Server:** `pr/video-manager-server.js` serves a local REST API and your `manage-videos.html` dashboard on `localhost:3000`.
3.  **The "Database":** You do not have a traditional database. Instead, your local database is literally the `src/components/Work.tsx` file. 
4.  **The Data Modification (The "Magic"):** When you add a video via the manager dashboard, the Express server reads `Work.tsx` as a raw string. It searches for `const projects = [...]` and `const reels = [...]` using Regular Expressions. It evaluates the array using `new Function()`, pushes the new video object to the array, serializes it back to a string, and forcibly overwrites the `Work.tsx` file on your hard drive.
5.  **The Deployment:** Immediately after modifying `Work.tsx`, the Node server uses `child_process.execFileSync` to `git add`, `git commit`, and `git push` the modified `Work.tsx` file to GitHub, triggering your hosting platform to rebuild the static site.

### 3. Modification Strategy
When customizing the UI and injecting new components, you must be extremely careful about **where** and **how** you edit the frontend to avoid breaking the local CMS.

*   **Where to inject components:** You can freely create new components in `src/components/` and style them with Tailwind. Modify `App.tsx` or `pages/Index.tsx` without worry.
*   **The Critical "Glue" (`src/components/Work.tsx`):** You can change the JSX/rendering code at the bottom of this file, but you must treat the top of the file as an immutable schema. 
*   **RULES FOR `Work.tsx`:** 
    *   Do **NOT** rename `const projects` or `const reels`.
    *   Do **NOT** move these arrays to a different file (like `data.ts`) unless you update the `WORK_FILE_PATH` and Regex in `video-manager-server.js`.
    *   Do **NOT** change the structural format of the arrays (e.g., extracting items into separate variables and pushing them into the array later). The Regex parser in your Node server expects a literal `const arrayName = [...]` structure.

### 4. Code Quality & Pitfalls (Technical Debt)
While the toolchain is clever, it introduces severe architectural bottlenecks that you should address before a massive overhaul.

**1. Regex & Eval AST Parsing (High Risk):**
Extracting and rewriting JavaScript arrays using Regex and `new Function("return [match]")` inside `video-manager-server.js` is incredibly fragile. 
*   *Pitfall:* If an IDE formatter (like Prettier/ESLint) modifies the wrapping of the array, or if you leave an unescaped character/comment inside the array in `Work.tsx`, the Regex will fail, the server will crash, and your site will be un-editable.
*   *Solution:* **Decouple your data from your components.** Move the `projects` and `reels` arrays into a sterile JSON file (e.g., `src/data/videos.json`). React can easily `import videoData from './videos.json'`, and your Node server can use native `JSON.parse` and `JSON.stringify` to update it, entirely eliminating the dangerous regex/eval logic.

**2. Blocking Synchronous Git Commands (Medium Risk):**
In `video-manager-server.js`, `execFileSync` is used to push to Git. 
*   *Pitfall:* This blocks the Node.js event loop. If your internet connection is slow or GitHub is delayed, the local web dashboard will completely freeze while the Express request hangs waiting for git to finish.
*   *Solution:* Refactor `execFileSync` to asynchronous `exec` or `spawn` so the Node server remains responsive, or return a 202 Accepted status bridging to a WebSocket/polling setup.

**3. Hardcoded Paths in Express (Minor):**
`WORK_FILE_PATH` relies on `path.join(__dirname, 'src/components/Work.tsx')`. If you restructure your `src` folder during your UI overhaul, remember to update this path in `video-manager-server.js`.
